# CS 饰品数据存储与缓存设计方案 (适配 2C2G 极简架构)

针对目标服务器资源高度受限（仅 2G 内存，40G NVMe 硬盘）且受制于 API 严格限频（通过 `https://private-api.csqaq.com` 进行单 IP 1次/秒，或企业版 5分钟 1次全量获取）的现状，本存储方案彻底摒弃了传统爬虫“全量无脑写库”的重模式，转向 **Redis 主导内存计算 + PostgreSQL (TimescaleDB) 轻量级时序归档** 的混合架构。

---

## 一、 核心设计原则

1.  **内存优先，算完即扔**：实时套利策略的运算全部在 Python/Go 内存与 Redis 中完成。
2.  **严格的生命周期 (TTL)**：不保存无尽的历史数据。K 线切片仅保留最近 7 天的分钟级数据，超期自动聚合或删除，保护 40G 的磁盘空间和 PG 缓存。
3.  **读写分离的错觉**：前端页面的实时刷新**完全不碰 PostgreSQL**，全量从 Redis 获取最新快照，保障低延迟。

---

## 二、 Redis 缓存架构 (实时心脏)

Redis 负责承载所有的实时热点数据，在 2G 内存限制下，预估占用 < 100MB。

### 1. 实时价格快照 (Hash)
*   **Key**: `items:latest_price`
*   **Value**: Field 为 `market_hash_name`，Value 为包含双端最低价、最高求购价、成交量的轻量 JSON 字符串。
*   **用途**: 前端 Dashboard 轮询的唯一数据源，极其轻量迅速。

### 2. 策略基准线数据 (Key-Value)
*   **Key**: `items:baseline:{hash_name}`
*   **Value**: 存储该饰品的“7日均价”、“前一小时平均成交量”等统计聚合值。
*   **用途**: 供内存计算引擎判断“巨量扫货”或“恐慌砸盘”异动时作为对比基准，每小时由后台异步更新一次。

### 3. 高优调度队列 (ZSET)
*   **用途**: 在单 IP **1次/秒** 的珍贵请求配额下，无法实现无脑的高频轮询。因此，系统只针对流动性极高、ROI 逼近红线、或初次快照触发异动警报的商品，调高其 Score 并优先出队，针对性消耗抓取与复核配额。
*   **实现计划**: 【待实现】在 Celery 任务队列中配置 `sync_csqaq_all_prices` (每 5 分钟)，同时维护一个独立的队列专供 1次/秒 的高优单品复核。

## 三、 PostgreSQL + TimescaleDB (冷温数据归档)

为了画 K 线和辅助后续的深度复盘，依然需要落盘。通过 TimescaleDB 的超表（Hypertable）进行管理，但**配置严酷的数据淘汰策略**。

### 1. 饰品元数据表 (`items_metadata`) - 静态表
记录所有 3 万件饰品的固化属性。
*   `market_hash_name` (主键)
*   `buff_goods_id`, `youpin_template_id`
*   `type`, `quality`, `exterior` (武器类型、磨损等)

### 2. 时序价格表 (`price_history`) - 超表
*   `time` (TIMESTAMP NOT NULL)
*   `market_hash_name` (VARCHAR)
*   `platform` (VARCHAR: buff/youpin)
*   `price`, `buy_order_price`, `volume`
*   **TimescaleDB 策略**:
    *   **按天分区 (Chunk)**：`chunk_time_interval => 1 day`
    *   **自动淘汰 (Retention)**：设定策略 `drop_chunks`，超过 7 天的原始明细数据直接 Drop，绝不挤占 40G 硬盘。
    *   **持续聚合 (Continuous Aggregate)**：通过物化视图生成 1小时/1天的 K 线汇总，供长期查阅。

### 3. 报警日志表 (`alert_logs`)
记录触发了买入信号的历史事件，用于后期回测我们的“异动策略”成功率。

---

## 四、 数据库防爆内存配置 (极重要)

针对 2G 内存服务器，必须修改 `postgresql.conf`，剥夺 PG 默认的内存贪婪特性：
*   `shared_buffers = 256MB` (限制 PG 只用小块内存，其余留给系统和爬虫)
*   `work_mem = 4MB` (限制排序和复杂查询内存)
*   `maintenance_work_mem = 64MB`
