# 全量饰品实时监控数据流水线 (2GB 内存极限优化版)

依托 `csqaq.com` 提供的企业级数据开放 API (`https://private-api.csqaq.com`)，根据最新频率限制（单 IP 1 次/秒，或企业接口 5 分钟 1 次全量获取），系统由原先的“高频无限制并发”重构为**“定时脉冲、单队列串行或全量快照”**的准实时监控架构。鉴于目标服务器配置受限（2核 2G 内存，100Mbps 充足下行带宽），本数据流水线在设计上极致压榨内存，确保在极低延迟下不发生 OOM（内存溢出）。

---

## 一、 系统架构核心目标

*   **稳定限频 (Rate-Limited)**：严格遵守 API 单 IP 1次/秒 的限制或 5分钟全量拉取，避免被封禁。
*   **高优聚焦 (Priority Focus)**：由于无法高频扫描 3 万活跃饰品，改为利用 5 分钟快照配合 1 秒单次核心饰品（如 50-100 款高流动性饰品）精细化轮询。
*   **内存不溢出 (OOM Prevention)**：通过严格的串行控制与阅后即焚机制，将进程常驻内存死死压在 200MB 以内。

---

## 二、 实时监控流水线设计

### 1. 生产者：严格限频的拉取或全量快照 (Strict Rate-Limited Fetching)
利用 `asyncio` 和 `aiohttp`，并引入强制的全局限速器 (Rate Limiter)。
*   **策略 A (企业接口全量主线)**：【待实现】在 `celery_tasks.py` 中编写 `sync_csqaq_all_prices` 任务，每 5 分钟调用一次 `/api/v1/goods/get_all_goods_info`，获取全站 3 万+ 饰品最新价格。
*   **策略 B (普通接口复核支线)**：【待实现】基于 `1次/秒` 的限频配额，仅对主线中发现的高优异动饰品进行定点 API 确认。

### 2. 计算引擎：内存中的“阅后即焚” (Match & Burn)
数据抓取后**不落盘**，直接在内存中撮合，计算完毕立刻销毁原始数据。
*   **极速提取**：拿到庞大的 JSON 响应后，立刻仅提取 `price`, `volume`, `buy_order_price`，组装为轻量的 `NamedTuple`。
*   **垃圾回收 (GC)**：将原始 JSON 字典的引用彻底丢弃，让 Python 的垃圾回收器瞬间介入，防止“垃圾堆积”。
*   **策略执行**：【待实现】在 `backend/services/arbitrage_service.py` 中落实《套利方案.md》的交易量过滤与断层过滤逻辑（注意：基于悠悠买 Buff卖的单向套利修正）。
*   **信号分发**：【待实现】触发报警的数据推入 Redis Pub/Sub，并由新增的 `notification_service.py` 发送到前端或 **QQ机器人/QQ频道**。

### 3. 消费者：有损队列与异步落盘 (Lossy Backpressure)
套利计算在内存中完成后，保留历史数据（用于 K 线与回测）的任务交由异步 Worker 负责。
*   **有损队列保护**：数据推入 `asyncio.Queue(maxsize=5000)`。如果在高并发下 PostgreSQL 写入速度跟不上，队列一旦积满，新的 K 线切片数据**直接丢弃 (Drop)**。
*   **丢卒保车原则**：核心报警已在内存中完成，丢弃的仅仅是历史归档记录。这确保了数据库卡顿时，报警雷达绝不宕机。
*   **批量 Insert**：每累计 1000 条有效切片，执行一次 DB 批量插入，极大降低数据库 IO 压力。

---

## 三、 防伪与复核机制

由于 API 数据更新频率（5-15 分钟聚合一次），极易遇到已经不存在的“幽灵”单子：
1.  **高贵请求复核**：遇到 `ROI > 20%` 的极佳数据（极可能是已被扫空的挂错价单），必须占用宝贵的“1次/秒”额度进行二次针对性 Fetch 查验。仅当二次验证依然存在才触发买入动作。
2.  **强制求购垫底校验**：无论纸面价差多大，必须校验 `最低卖价 > 对方最高求购价` 的 Spread，过滤掉无人问津的“死水空盘”。
